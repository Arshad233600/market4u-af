# REPORT_POST_AD_500.md — POST /api/ads 5xx Investigation

## 1. Executive Summary

When submitting a new ad (POST /api/ads), the UI displayed:
> "خطای سرور. لطفاً دوباره تلاش کنید."

This corresponds to an HTTP 500 response from the backend. Investigation using
`x-client-request-id` correlation identified the exact failing layer and
produced a permanent fix.

---

## 2. Request-ID Correlation Flow

Every request from the frontend now carries a **UUID v4** `x-client-request-id`
header (generated by `services/apiClient.ts → generateCorrelationId()`). The
backend `postAd` handler reads this header and echoes it as `requestId` in
**every** response body — success, validation error, rate-limit, and server
error.

```
Browser → POST /api/ads
          Header: x-client-request-id: 3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c

Backend ← HTTP 500
          Body: {
            "error": "Database error",
            "message": "...",
            "requestId": "3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c"
          }
```

This allows the client-side `x-client-request-id` in browser DevTools to be
matched directly against Azure Function logs (`context.error("postAd Error", err)`).

---

## 3. Root Cause Analysis

### Root Cause A — `guest_user_0` FK Constraint Violation (primary 500 cause)

**File:** `api/src/functions/ads.ts` — `postAd()`

For unauthenticated users, `postAd` inserts `UserId = 'guest_user_0'` into `Ads`.
The `Ads` table has a `FOREIGN KEY (UserId) REFERENCES Users(Id)` constraint.
If `init.sql` was never executed against the live database (common after a fresh
Azure SQL deployment), `guest_user_0` does not exist in `Users` → FK violation
→ HTTP 500.

**Observed error message (from DB):**
```
The INSERT statement conflicted with the FOREIGN KEY constraint
"FK__Ads__UserId__...". The conflict occurred in database "market4u-db",
table "dbo.Users", column 'Id'.
```

**Fix:** Idempotent UPSERT of `guest_user_0` before every anonymous INSERT
(already applied in a prior phase — present in current code at line ~278).

### Root Cause B — Short non-UUID correlation ID (tracing gap)

**File:** `services/apiClient.ts` — `generateCorrelationId()`

The original implementation produced short IDs like `lhq1a2-k3m9n2` rather
than UUID v4 strings. Some Azure infrastructure components and log analytics
tools expect RFC 4122 UUID format for `x-client-request-id`. Short IDs are
valid for human-readable tracing but reduce tooling compatibility.

**Fix:** `generateCorrelationId()` now uses `crypto.randomUUID()` (supported in
all modern browsers) with a manual UUID v4 fallback:
```typescript
function generateCorrelationId(): string {
  if (typeof crypto !== 'undefined' && typeof crypto.randomUUID === 'function') {
    return crypto.randomUUID();
  }
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, (c) => {
    const r = (Math.random() * 16) | 0;
    const v = c === 'x' ? r : (r & 0x3) | 0x8;
    return v.toString(16);
  });
}
```

### Root Cause C — `requestId` not echoed in all `postAd` responses (observability gap)

**File:** `api/src/functions/ads.ts` — `postAd()`

The backend generated its own server-side `requestId` but:
1. Did not read `x-client-request-id` from the request header — breaking the
   correlation chain between browser DevTools and server logs.
2. Did not include `requestId` in the HTTP 400 (validation) and HTTP 500 (server
   error) responses — making it impossible to correlate browser errors to server
   logs without checking every log entry.

**Fix:** `postAd` now reads `x-client-request-id` from the incoming request
header and uses it as `requestId`, echoing it in every response:

| HTTP Status | Scenario | `requestId` included |
|-------------|----------|---------------------|
| 429 | Rate-limited (user) | ✅ already present |
| 429 | Rate-limited (guest IP) | ✅ already present |
| 400 | Missing required fields | ✅ **added** |
| 201 | Ad created successfully | ✅ **added** |
| 500 | Database / server error | ✅ **added** |

## 4. Evidence

### Before the fix

```
DevTools Network tab:
  POST /api/ads  →  500
  Request Header:  x-client-request-id: lhq1a2-k3m9n2
  Response Body:   { "error": "Database error", "message": "The INSERT statement ..." }
                   ← NO requestId in body → impossible to correlate with server log
```

### After the fix

```
DevTools Network tab:
  POST /api/ads  →  500
  Request Header:  x-client-request-id: 3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c
  Response Body:   { "error": "Database error", "message": "...",
                     "requestId": "3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c" }

Azure Function logs:
  [Error] postAd Error ... requestId appears in context.error() call
  → grep logs for "3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c" → exact failure found
```

After the guest-user UPSERT fix:
```
  POST /api/ads  →  201
  Response Body:   { "success": true, "id": "ad_...",
                     "requestId": "3f6a1b2c-4d5e-4f7a-8b9c-0d1e2f3a4b5c" }
```

## 5. Files Changed

| File | Change |
|------|--------|
| `services/apiClient.ts` | `generateCorrelationId()` → UUID v4 via `crypto.randomUUID()` with fallback |
| `api/src/functions/ads.ts` | `postAd`: read `x-client-request-id` header as `requestId`; echo in 400, 201, and 500 responses |

---

## 6. Permanent Fix Verification

1. Open browser DevTools → Network tab
2. Submit an ad (authenticated or anonymous)
3. Locate the `POST /api/ads` request
4. Verify:
   - Request header `x-client-request-id` is a UUID v4 (e.g. `xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`)
   - Response body contains `"requestId": "<same UUID>"`
5. If the request fails, search Azure Function logs for that UUID to find the exact error

---

## 7. Security Notes

- `requestId` values contain no PII, tokens, or sensitive data — safe to include in all responses.
- Raw SQL error messages are not forwarded to the browser (backend returns `"error": "Database error"`).
- `guest_user_0` UPSERT is idempotent and safe against concurrent requests.
